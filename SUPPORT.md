# Getting Support

Thank you for using the STARTcloud Provisioner Catalog! This document outlines
the ways you can get help.

## Available Resources

### Documentation

- **[README](README.md)** — what the catalog is, the URL agents fetch, the
  consumer contract, and both publishing doors
- **[CONTRIBUTING](CONTRIBUTING.md)** — the submitter onboarding guide:
  package requirements, the validation action, and the admission flow
- **[examples/](examples/)** — the copy-paste publisher kit for new
  provisioner repositories

### Issues and Bug Reports

- **[GitHub Issues](https://github.com/STARTcloud/provisioner-catalog/issues)** —
  report bugs (stale or wrong catalog data, validation problems) or request
  features
- Use the appropriate issue template and search existing issues first to
  avoid duplicates

### Community Discussion

- **[GitHub Discussions](https://github.com/STARTcloud/provisioner-catalog/discussions)** —
  ask questions and share ideas

## Support Expectations

The catalog is maintained with limited development resources:

- **Response time**: usually a few days, but this varies with workload and
  complexity
- **Priority**: catalog integrity issues (tripwire, checksums) and admission
  PRs receive the highest priority
- **Feature requests**: evaluated against community need and available time

## Getting Quality Help

### For Bug Reports

1. **Use the bug report template**
2. **Identify the component** — published catalog.json, the data job, the
   validation action, PR checks, or the schemas
3. **Include specifics** — the affected repository/family/version, links to
   failing workflow runs, and exact error output
4. **Check the data-job cadence first** — catalog.json refreshes every ~2
   hours; data newer than that is not yet published

### For Questions

1. **Check the README and CONTRIBUTING first**
2. **Search existing issues and discussions**
3. **Be specific** about what you're trying to achieve

## Contributing Back

The best way to improve support for everyone:

- **Document solutions** you find to help others
- **Contribute fixes** when you solve issues
- **Improve documentation** based on your onboarding experience
- **Help answer questions** from other publishers

See our [Contributing Guide](CONTRIBUTING.md) for more information.
